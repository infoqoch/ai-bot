## 1. scheduler.py - CLAUDE.md에 "세션 compact 스케줄러"라 했는데 /scheduler에 안나옴

### 현재 상태 분석

- `src/scheduler.py`의 `SessionScheduler`는 **매니저 세션 자동 compact** 목적으로 만들어짐
- 그런데 코드 내부에 `logger.info("SessionScheduler: 매니저 세션 제거로 인해 스케줄 작업 없음")`이라 적혀 있음
- 즉, 매니저 세션 기능이 제거되면서 **실제로 아무 작업도 등록하지 않는 빈 껍데기**
- `/scheduler` 명령어는 `SchedulerManager`에 등록된 job만 보여주는데, `SessionScheduler`는 아무것도 등록 안 함

### 의견

- `SessionScheduler`는 **죽은 코드**. 삭제 대상
- CLAUDE.md의 `scheduler.py # 세션 compact 스케줄러` 설명도 함께 제거 또는 수정
- 스케줄 관련 모듈은 이미 `SchedulerManager` + `ScheduleService` + `ScheduleAdapter`로 통합되어 있음
- `scheduler.py` 파일 자체를 삭제하고 CLAUDE.md 구조도를 업데이트하면 됨

### 개발 계획

| 순서 | 작업 | 파일 |
|------|------|------|
| 1 | `src/scheduler.py` 삭제 | scheduler.py |
| 2 | `src/main.py`에서 SessionScheduler import/사용 제거 | main.py |
| 3 | CLAUDE.md 구조도에서 `scheduler.py` 항목 제거 | CLAUDE.md |

## 3. 플러그인 schema.sql 관리 - 플러그인이 봇 스키마에 직접 추가하는 것은 부적절

### 현재 상태 분석

- 현재 `schema.sql`에 `memos`, `todos`, `weather_locations` 테이블이 **직접 포함**되어 있음
- 빌트인이든 커스텀이든 모든 플러그인이 봇의 핵심 스키마에 직접 테이블을 추가하는 구조
- `Repository`에 `get_weather_location()`, `add_todo()` 등 플러그인 전용 메서드가 있음

### 의견

**동의: 플러그인 스키마 분리가 필요**

현재 구조의 문제:
1. 플러그인 추가/제거 시 봇의 핵심 스키마를 건드려야 함
2. 커스텀 플러그인이 메인 코드를 수정해야 함 → 업그레이드 시 충돌
3. 빌트인도 마찬가지 - 빌트인 플러그인 제거 시 schema.sql에 죽은 테이블 잔존

**제안 구조:**

```
plugins/
├── builtin/
│   └── todo/
│       ├── plugin.py
│       ├── schema.sql        ← 플러그인 자체 스키마
│       └── repository.py     ← 플러그인 자체 데이터 접근
│
└── custom/
    └── my_plugin/
        ├── plugin.py
        ├── schema.sql        ← 커스텀 플러그인 스키마
        └── repository.py
```

**스키마 로딩 방식:**
- 봇 시작 시 `init_schema()`가 메인 schema.sql 실행
- 이후 각 플러그인 디렉토리의 schema.sql을 자동 실행
- 테이블 이름에 prefix 부여: `plugin_{name}_` (예: `plugin_todo_items`)
  - 단, 베타 단계에서 이 prefix는 강제가 아닌 **권장**으로 두는 것이 현실적

**빌트인도 동일하게 적용?**
- YES. 빌트인도 결국 "플러그인"이므로 동일 규칙 적용이 맞음
- 다만 빌트인은 메인 Repository를 공유할 수 있는 특권을 줄 수 있음 (편의성)

**충돌 방지:**
- 테이블 이름 prefix 권장 (`plugin_{name}_`)
- 명령어 중복 감지: `PluginLoader.load_all()` 시 `name` 중복 체크하여 경고
- 콜백 prefix 중복 감지: `CALLBACK_PREFIX` 충돌 시 로그 경고

**실제 구현 비용은 크지 않음.** 하지만 베타 단계에서는 빌트인만 존재하므로 **기획만 해두고 베타 이후에 구현**하는 것도 합리적.

### 개발 계획 (베타 이후 권장, 지금은 설계만)

| 순서 | 작업 | 비고 |
|------|------|------|
| 1 | 플러그인별 schema.sql 로딩 구조 설계 | PluginLoader에 스키마 로드 추가 |
| 2 | 빌트인 플러그인 스키마 분리 (memos, todos, weather_locations) | schema.sql → 각 플러그인 폴더 |
| 3 | 플러그인 자체 Repository 패턴 정의 | Plugin 클래스에 `_init_db()` |
| 4 | 이름/콜백 prefix 충돌 감지 로직 추가 | PluginLoader.load_all() |
| 5 | CLAUDE.md 플러그인 개발 가이드 업데이트 | Layer 2 |

## 4. 콜백 prefix 테이블 - 플러그인마다 CLAUDE.md를 수정하면 안 됨

### 현재 상태 분석

- `callback_handlers.py`에서 prefix별로 if-else 분기 처리
- 새 플러그인 추가 시 이 파일에 수동으로 분기 추가 필요
- CLAUDE.md에도 수동으로 prefix 테이블 업데이트 필요

### 의견

**동의: 자동 라우팅이 필요**

현재 패턴:
```python
if callback_data.startswith("td:"):
    await self._handle_todo_callback(...)
elif callback_data.startswith("memo:"):
    await self._handle_memo_callback(...)
```

개선 패턴:
```python
# PluginLoader가 CALLBACK_PREFIX 기반으로 자동 라우팅
for plugin in self.plugins:
    if hasattr(plugin, 'CALLBACK_PREFIX') and callback_data.startswith(plugin.CALLBACK_PREFIX):
        result = await plugin.handle_callback_async(callback_data, chat_id)
        break
```

이렇게 하면:
- CLAUDE.md에 prefix 테이블 수동 관리 불필요
- callback_handlers.py에 플러그인 분기 추가 불필요
- 플러그인이 자기 폴더 안에서 완결

### 개발 계획

| 순서 | 작업 |
|------|------|
| 1 | Plugin 클래스에 `CALLBACK_PREFIX` 정식 속성 추가 |
| 2 | CallbackHandlers에서 플러그인 자동 라우팅 구현 |
| 3 | 빌트인 플러그인의 콜백 핸들링을 각 plugin.py로 이관 |
| 4 | CLAUDE.md에서 prefix 테이블을 "자동 관리됨" 으로 변경 |

## 7. 4096자 제한 - 분할 전송 메커니즘

### 현재 상태 분석

**4096은 텔레그램 Bot API의 메시지 최대 길이 제한이다.**
- 텔레그램 `sendMessage`의 `text` 필드는 1~4096자 제한 (UTF-8)
- 초과 시 API 에러 발생: `MESSAGE_TOO_LONG`

**현재 봇의 처리 방식:**
```python
# base.py _send_message_to_chat()
async def _send_message_to_chat(self, bot, chat_id, text, max_length=4000):
    if len(text) <= max_length:
        await bot.send_message(...)
        return

    # 4000자씩 자르고 여러 메시지로 분할 전송
    chunks = [text[i:i + max_length] for i in range(0, len(text), max_length)]
    for chunk in chunks:
        await bot.send_message(...)
```

- **안전 마진 4000자**: HTML 태그가 포함되면 실제 텍스트보다 길어지므로 96자 여유
- **분할 전송**: 4000자 초과 시 여러 메시지로 나눠 보냄
- **Claude에는 별도 제한 없음**: Claude CLI는 원하는 만큼 응답 가능

### 의견

**현재 구조는 합리적이나, 개선 포인트 있음:**

1. **현재 분할 방식의 문제**: 단순히 글자 수로 자르면 HTML 태그가 중간에 잘릴 수 있음
   - `<b>text가 여기서 잘리면</b>` → `<b>text가 여기서` + `잘리면</b>` → 파싱 에러
   - 현재는 HTML 파싱 실패 시 plain text로 fallback하긴 함

2. **개선안**:
   - **A안**: 분할 시 HTML 태그 안전 경계(줄바꿈) 기준으로 자르기
   - **B안**: 4000자 초과 시 텍스트 파일로 전송 (document)
   - **C안**: 현재 유지 (실용적으로 큰 문제 없음, HTML fallback이 있으므로)

3. **Claude 응답 길이 제한?**
   - Claude에게 "4000자 이내로" 지시하는 것은 **비효율적** (프롬프트에 넣어도 정확히 지키지 못함)
   - 봇이 사후 처리(분할/파일 전송)하는 현재 방식이 맞음

**추천: C안 (현재 유지) + 장기적으로 A안 보완**
- 현재 fallback이 있어 실용적 문제가 거의 없음
- 완벽한 해결은 줄바꿈 기준 분할이지만 우선순위 낮음

### 개발 계획 (낮은 우선순위)

| 순서 | 작업 |
|------|------|
| 1 | `_send_message_to_chat()`에서 줄바꿈 기준 분할로 개선 |
| 2 | SPEC.md 표현 수정: "안전 마진 4000자, 초과 시 자동 분할 전송" 유지하되 근거 명시 |

## 9. 히스토리 처리자 표시 `[cmd]`, `[plugin]`, `[x]` 설명

```python
# message_handlers.py
self._temp_pending = {  # ← 딕셔너리 하나로 관리!
    "user_id": ...,
    "message": ...,
    ...
}
```

**문제 확인: `_temp_pending`이 단일 딕셔너리!**
- 두 번째 충돌 메시지가 오면 첫 번째를 **덮어씀**
- 첫 번째 메시지의 버튼을 눌러도, `_temp_pending`에는 두 번째 데이터만 있음 → **엉뚱한 메시지 처리**

### 의견

**확인된 버그.** `_temp_pending`을 메시지별로 관리해야 함.

개선안:
```python
# 콜백 데이터에 메시지 해시를 포함
self._temp_pending: dict[str, dict] = {}  # key = 메시지 해시
```

또는 `SessionQueueManager`의 `add_to_waiting()`에 이미 큐가 있으므로, `_temp_pending` 대신 큐를 직접 활용.

### 개발 계획

| 순서 | 작업 |
|------|------|
| 1 | `_temp_pending`을 딕셔너리 → 키 기반 매핑으로 변경 |
| 2 | 콜백 데이터에 고유 ID 포함하여 올바른 메시지 매핑 |
| 3 | 다중 충돌 시나리오 테스트 |

## 16. /lock → /tasks 리네이밍 + 세션리스트 연동

### 현재 상태 분석

- `/lock` = 현재 처리 중인 Claude 태스크 + 대기열 현황
- `/sl` (session_list) = 세션 목록
- 세션 목록에 `[Tasks]` 버튼이 있어 `/lock` 화면으로 이동
- `/lock` 화면에는 `[Refresh]` 버튼만 있음

**`/lock`과 `/sl`의 관계:**
- `/sl`에서 `[Tasks]` 버튼 → `/lock` 화면
- `/lock`에서 `/sl`로 돌아가는 버튼은 **없음** → **편의성 부족**

### 의견

1. **리네이밍 `/lock` → `/tasks`**: 동의. "lock"은 내부 구현 용어, "tasks"가 사용자 관점에서 자연스러움
2. **`[Session List]` 버튼 추가**: 동의. 양방향 네비게이션 필요
3. **`/lock`은 별칭으로 유지**: 호환성

### 개발 계획

| 순서 | 작업 |
|------|------|
| 1 | `/tasks` 명령어 추가 (= `/lock` 별칭) |
| 2 | tasks 화면에 `[Session List]` 버튼 추가 |
| 3 | SPEC.md, CLAUDE.md, /help 텍스트 업데이트 |
| 4 | `/lock`은 별칭으로 유지 |
